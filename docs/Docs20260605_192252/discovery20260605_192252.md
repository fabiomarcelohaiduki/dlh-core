## Visao

### Problema
A operacao do grupo DLH depende de pessoas-chave e de conhecimento disperso (cabeca de funcionarios, e-mails, conversas). Quando alguem sai, viaja ou se ausenta, processos travam, decisoes se perdem e a memoria operacional evapora. Falta um ecossistema unico que centralize fontes (portais, ERP, e-mails, drive), persista memoria viva e automatize a operacao de forma continua, sem refem de individuos.

### Usuario principal
Nucleo operacional do grupo DLH:
- **Fábio**: lideranca e decisoes estrategicas.
- **Ligiane (esposa/socia)**: co-lideranca da operacao.
- **Funcionarios**: operadores do dia a dia.
- **Lia (IA)**: agente que opera dentro do LionClaw, intermediando o DLH Core com as pessoas.

Time pequeno, alta confianca, cada pessoa acumula varias responsabilidades. O DLH Core serve humanos e Lia sem fricao.

### Referencia
**SOM (Symbiotic Operating Model)** — framework autoral do fundador. O DLH Core e a primeira materializacao do SOM em software. Principios estruturais:
- Tres atores de primeira classe: empresa, IA (Lia) e pessoas.
- Niveis de autonomia configuraveis (de execucao assistida a bloqueada para acoes sensiveis).
- Memoria operacional persistente como substrato, nao log.
- Redundancia humano <-> IA: se um falta, a operacao segue.
- Rastreabilidade e governanca embutidas.

**MOE (Memoria Operacional Evolutiva)** — camada de memoria do SOM, tambem autoral:
- Memoria viva e evolutiva, nao arquivo morto.
- Distingue fato, interpretacao, hipotese, decisao, aprendizado.
- Governada, rastreavel, com niveis de risco e sensibilidade.
- Privilegia memoria util sobre acumulo.

**Natureza:** DLH Core e o **ecossistema inteiro** (humanos + sistemas + canais + Lia), nao um produto. Funciona como **loop continuo**: fontes -> nosso build (cockpit + pipeline LLM-ready + substrato Supabase) -> Lia (cerebro) -> automacoes -> LionClaw materializa -> sistemas destino -> novos dados realimentam o loop = aprendizado continuo (MOE).

**O que esta sendo construido agora:** o primeiro componente do ecossistema — **Substrato + Cockpit de coleta**, com a primeira fonte (Effecti) integrada e a Lia consumindo via API LLM-ready. Nao e o DLH Core inteiro.

**Principio de tratamento de dados: LLM-ready by design.** Tudo normalizado, enriquecido, com embeddings e contexto pronto para a Lia consumir como cerebro — nao JSON cru.

### Pitch
O DLH Core e o nucleo operacional simbiotico do grupo DLH: uma plataforma que materializa o SOM fazendo empresa, pessoas e IA (Lia, operando dentro do LionClaw) compartilharem conhecimento, execucao, memoria e validacao numa mesma estrutura viva.

Comeca capturando avisos e editais do portal Effecti via API, ja tratados em formato LLM-ready. Depois evolui para e-mails, ERP Nomus, Google Drive e automacoes setoriais — sempre com niveis de autonomia explicitos e uma MOE que distingue fato, interpretacao, hipotese e decisao.

Resultado: uma operacao que nao trava quando alguem se ausenta — contexto, decisoes e proximas acoes ficam persistidos, rastreaveis e executaveis pela Lia dentro das regras definidas.

## Funcionalidades

### Core features

**1. Conector Effecti + pipeline LLM-ready**
Coleta avisos e editais do portal Effecti via API. Pipeline trata o dado antes de gravar no Supabase: agendamento configuravel + execucao sob demanda, janelas de sincronizacao, sync incremental, versionamento de mudancas (errata/prorrogacao/cancelamento), enriquecimento cognitivo (campos-chave, resumo, classificacao, embeddings, fato vs hipotese) e arquitetura extensivel para os proximos conectores.

**2. Substrato de memoria simbiotica (Supabase)**
"Cerebro" persistente do ecossistema. Schema modelado pela MOE (fato/interpretacao/hipotese/decisao/aprendizado, rastreabilidade, confiabilidade, sensibilidade). Busca semantica por embeddings, governanca por sensibilidade, historico/versionamento. Acessivel por humanos (UI) e pela Lia (contexto pronto, nao SQL bruto).

**3. Cockpit unificado**
Interface unica que consolida: visao humana dos dados coletados, administracao da ingestao (agendamentos, fontes, credenciais), monitoramento operacional (status, erros, healthcheck) e API/MCP LLM-ready para a Lia consumir o substrato. Mesmo backend para humanos e Lia.

**Faseamento:**
- **Fase 1 (MVP):** coletar + centralizar.
- **Fase 2:** analisar + cruzar (e-mails, Nomus, Drive, automacoes setoriais).

### Integracoes
**MVP (Fase 1):**
- **API portal Effecti** — primeira fonte oficial. Credenciais e documentacao da API ja disponiveis (endpoints, paginacao, rate limit, webhooks).
- **Supabase** — substrato unico (Postgres + Auth + Storage + Edge Functions).
- **LionClaw** — consumidor da API LLM-ready do cockpit (a Lia opera dentro do LionClaw).
- **Claude (Anthropic)** — provedor de LLM do pipeline de ingestao (resumos, classificacao, extracao, embeddings). **Restricao: SEM API paga por token. Consumo via plano Claude Max ja contratado** (mesmo da Lia/LionClaw). Decisao de "como" acessar o Claude (via Lia/LionClaw, sem api).
- **Agendador / fila**: Supabase Edge Functions + pg_cron (mesmo substrato, menos pecas moveis).
- **Observabilidade**: Sentry (erros e exceptions) + logs nativos do Supabase (execucoes, syncs, audit trail).

## Tecnico

- Sentry (erros) + logs nativos Supabase (execucoes, audit trail).
